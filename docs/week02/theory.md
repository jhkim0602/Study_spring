# Week 02 Theory

## 주제

Spring MVC의 가장 기본적인 흐름과 JSP 뷰 처리를 이해한다.

## 이 문서를 읽기 전에

2주차 이론은 "스프링이 내부적으로 얼마나 복잡한가"를 배우는 문서가 아니다.  
오히려 아주 기본적인 질문에 답하는 문서다.

- 브라우저가 URL을 요청하면 누가 받는가?
- 자바 코드는 어디서 실행되는가?
- JSP는 HTML과 무엇이 다른가?
- 컨트롤러가 왜 필요한가?

이 질문들이 정리되어야 이후의 Bean, DI, Scope도 자연스럽게 이해된다.

## 1. 웹 애플리케이션이란 무엇인가

우리가 만드는 것은 콘솔 프로그램이 아니라 웹 애플리케이션이다.

웹 애플리케이션의 가장 단순한 구조:

```text
사용자 브라우저
  -> 요청
  -> 서버
  -> 처리
  -> HTML 응답
  -> 브라우저 화면 표시
```

이 프로젝트에서는:

- 브라우저: 사용자의 웹 브라우저
- 서버: Spring Boot가 띄우는 내장 Tomcat
- 처리 코드: Controller와 자바 클래스
- 화면: JSP

## 2. 요청과 응답

### 요청(Request)

브라우저가 서버에 보내는 메시지다.

예:

- `/`
- `/hello`
- `/week04/configDI`

같은 URL 요청

### 응답(Response)

서버가 브라우저에 돌려주는 결과다.

예:

- HTML 문서
- JSON 데이터
- 에러 페이지

2주차에서는 주로 HTML 화면 응답을 본다.

## 3. MVC는 왜 필요한가

MVC는 역할을 나누기 위한 구조다.

### Model

화면에 전달할 데이터다.

예:

- 사용자 이름
- 문자 발신 결과
- 빈 정보

### View

사용자에게 보여줄 화면이다.

현재 프로젝트에서는 JSP가 View 역할을 한다.

### Controller

요청을 받고,
어떤 데이터를 준비해서,
어떤 화면으로 보낼지 결정하는 역할이다.

왜 MVC가 필요한가:

- 화면 코드와 자바 로직이 덜 섞인다
- 유지보수가 쉬워진다
- 코드 역할이 분명해진다

## 4. JSP는 무엇인가

JSP는 `Java Server Pages`의 약자다.

핵심 아이디어:

- HTML 안에 서버 쪽 동적 표현을 섞을 수 있다
- 서버에서 JSP를 처리해 최종 HTML로 바꾼 뒤 브라우저에 보낸다

즉 브라우저가 JSP 파일 자체를 받는 것이 아니라,
서버가 JSP를 실행한 결과인 HTML을 받는다.

## 5. Spring MVC에서 누가 요청을 받는가

스프링 MVC에는 요청을 중앙에서 받는 핵심 컴포넌트가 있다.

이름은 `DispatcherServlet`이다.

역할:

- 들어온 요청을 받는다
- 어떤 컨트롤러가 처리할지 찾는다
- 컨트롤러 실행 결과를 바탕으로 뷰를 찾는다

쉽게 말하면:

> Spring MVC의 교통정리 담당

이라고 보면 된다.

## 6. Controller는 왜 필요한가

만약 컨트롤러가 없다면,

- 어떤 URL을 어떤 자바 코드가 처리하는지
- 어떤 데이터를 화면에 보낼지
- 어떤 JSP를 보여줄지

를 체계적으로 관리하기 어렵다.

컨트롤러는 이 중간 연결을 담당한다.

예를 들어:

```java
@GetMapping("/hello")
public String hello() {
    return "week02/hello";
}
```

이 코드는:

- `/hello` 요청을 받고
- `week02/hello`라는 뷰 이름을 반환하는 의미다

## 7. View Resolver는 왜 필요한가

컨트롤러는 보통 JSP의 전체 경로를 직접 쓰지 않는다.

예:

```java
return "week02/hello";
```

이 문자열은 "뷰 이름"이지 실제 파일 경로가 아니다.

실제 파일 경로는 View Resolver가 만든다.

현재 프로젝트의 설정을 기준으로 하면:

- prefix: `/views/`
- suffix: `.jsp`

따라서:

- 뷰 이름: `week02/hello`
- 실제 경로: `/views/week02/hello.jsp`

가 된다.

## 8. 왜 JSP 파일만 있어서는 안 되는가

초보자가 가장 많이 하는 오해가 이것이다.

> "JSP 파일이 있으니까 URL로 바로 열리겠지?"

하지만 보통은 그렇지 않다.

이유:

- JSP는 화면 파일일 뿐이다
- URL과 JSP 파일이 자동 연결되는 것은 아니다
- 누가 어떤 JSP를 보여줄지 컨트롤러와 뷰 리졸버가 정해야 한다

즉:

- JSP = 화면 조각 또는 화면 파일
- Controller = 요청 처리자
- View Resolver = 화면 파일 경로 연결자

다.

## 9. 현재 프로젝트에서 왜 `src/main/webapp/views/`를 쓰는가

이 위치는 JSP를 두기 위한 전형적인 위치다.

구조 예:

```text
src/main/webapp/
  index.jsp
  views/
    week02/
      hello.jsp
```

왜 `views/` 아래에 두는가:

- 주차별 화면을 구분하기 쉽다
- View Resolver 설정과 맞추기 쉽다
- 화면 파일을 한 곳에서 관리할 수 있다

## 10. 2주차 개념이 이후 주차와 어떻게 연결되는가

2주차를 이해하면 이후 주차에서 이런 연결이 보인다.

- 3주차: 컨트롤러가 컨테이너에서 빈을 꺼내 뷰로 전달
- 4주차: DI된 객체를 컨트롤러에서 확인
- 5주차: scope/lifecycle 결과를 JSP로 출력
- 6주차: 실습 결과를 다시 JSP로 보여줌

즉 이후 주차가 어려워 보여도,
결국 출력 구조는 계속 MVC다.

## 11. 자주 헷갈리는 질문

### Q1. JSP는 자바 클래스인가?

직접 작성할 때는 HTML 기반 템플릿처럼 보이지만,
서버가 내부적으로 처리할 때는 자바 코드로 변환되어 실행된다.

### Q2. HTML만 쓰면 안 되나?

정적인 화면만 필요하면 HTML만으로도 가능하다.  
하지만 서버 데이터에 따라 내용이 달라져야 하면 JSP 같은 서버 뷰 기술이 필요하다.

### Q3. Controller가 데이터를 직접 화면에 출력하면 안 되나?

가능은 하지만 역할이 섞여 구조가 나빠진다.  
그래서 Controller는 처리와 데이터 준비, View는 화면 생성에 집중시키는 것이 좋다.

## 12. 기억할 핵심 문장

> 브라우저 요청은 컨트롤러가 받고, 컨트롤러는 뷰 이름을 반환하며, View Resolver는 그 이름을 실제 JSP 경로로 연결한다.
