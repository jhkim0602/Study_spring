# Week 03

## 주제

스프링 컨테이너와 Bean 개념 이해, XML 설정 방식과 Java 설정 방식의 차이 정리.

## 현재 원칙

이번 주차 내용은 아직 프로젝트 실행 코드에 적용하지 않고 문서로만 정리한다.

## 핵심 개념

### 1. 스프링 컨테이너

- 객체를 생성하고 관리하는 역할을 한다.
- 개발자가 직접 `new`를 반복하기보다, 스프링에게 객체 생성을 맡긴다.
- 스프링이 관리하는 객체를 Bean이라고 부른다.

### 2. Bean

- 스프링 컨테이너 안에 등록되어 관리되는 객체다.
- 필요할 때 컨테이너에서 꺼내 사용한다.

### 3. `@Configuration`

- 이 클래스가 스프링 설정 클래스라는 뜻이다.
- "어떤 Bean을 만들지 적어두는 설계도"라고 이해하면 된다.

### 4. `@Bean`

- 메서드가 반환하는 객체를 스프링 컨테이너에 등록하라는 뜻이다.
- 기본 Bean 이름은 메서드 이름을 따른다.

예시:

```java
@Configuration
public class AppConfig {

    @Bean
    public SmsSender smsSender() {
        return new SmsSender();
    }
}
```

의미:

- `AppConfig`는 설정 클래스
- `smsSender()`가 반환한 객체는 Bean 등록 대상
- 컨테이너에서 `smsSender` 이름으로 꺼낼 수 있다

## XML 방식과 Java 방식 비교

### XML 기반 설정

장점:

- 설정과 자바 코드를 분리해서 볼 수 있다.

아쉬운 점:

- 초보자 입장에서는 코드와 설정이 분리되어 흐름을 따라가기 어렵다.
- 타입 확인과 자동완성이 자바 기반 설정보다 불편할 수 있다.

### Java 클래스 기반 설정

장점:

- IDE 지원이 좋다.
- 타입 오류를 더 빨리 확인할 수 있다.
- 코드와 설정 흐름을 한 번에 보기 쉽다.

## 왜 중요한가

- 객체 생성을 한 곳에서 관리할 수 있다.
- 나중에 의존관계가 많아져도 구조를 통제하기 쉽다.
- 유지보수와 실습 확장이 편해진다.

## 이번 주차에서 기억할 문장

"스프링 컨테이너는 객체를 관리하고, 그 안에서 관리되는 객체를 Bean이라고 한다."

"`@Configuration`은 설정 클래스 표시이고, `@Bean`은 객체 등록 지점이다."
