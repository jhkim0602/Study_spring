## 1. 6주차 큰 흐름

PPT는 아래 질문들을 중심으로 전개된다.

- 빈은 얼마나 오래 살아남는가
- 같은 이름의 빈을 두 번 찾으면 같은 객체인가
- 웹 요청과 세션에 따라 객체 범위가 달라지는가
- 빈이 생성되고 종료될 때 어떤 메서드가 호출되는가
- 빈이 자기 이름이나 컨테이너를 직접 알아야 할 때는 어떻게 하는가
- 설정값을 코드 밖으로 빼면 어떻게 읽는가

즉 6주차는 "빈 등록"이 아니라 "빈 운영" 주차라고 볼 수 있다.

## 2. 스코프 4가지

### 2-1. singleton

- 기본 스코프
- 컨테이너 안에 인스턴스 하나만 존재
- 여러 요청과 여러 객체가 같은 인스턴스를 공유

### 2-2. prototype

- 필요할 때마다 새 객체 생성
- `getBean()`을 호출할 때마다 다른 객체를 받는다

### 2-3. request

- HTTP 요청 하나 동안만 유지
- 요청이 끝나면 더 이상 유지되지 않는다

### 2-4. session

- 사용자 세션마다 하나 유지
- 같은 브라우저 세션에서는 재사용된다

## 3. 서로 다른 범위 빈 의존 문제

PPT 7~8장의 핵심이다.

### 문제

singleton 빈이 prototype 빈을 생성자나 필드로 직접 주입받으면  
singleton이 만들어질 때 딱 한 번 생성된 prototype을 계속 사용한다.

즉 선언은 prototype이어도 실제 사용은 거의 singleton처럼 고정된다.

### 해결

`ObjectFactory<T>`를 주입받고 `getObject()`를 호출한다.

이 방식은 "빈이 필요할 때 그때 컨테이너에서 새로 꺼내기" 때문에  
prototype이나 request 빈과 함께 쓸 때 유용하다.

## 4. 라이프사이클 메서드

PPT 9~10장의 범위다.

확인해야 할 훅은 4종류다.

### 4-1. 어노테이션 방식

- `@PostConstruct`
- `@PreDestroy`

### 4-2. 인터페이스 방식

- `InitializingBean.afterPropertiesSet()`
- `DisposableBean.destroy()`

### 4-3. 커스텀 메서드 방식

- `@Bean(initMethod = "...", destroyMethod = "...")`

정리하면,

- 생성자 이후
- 의존성 설정 완료 이후
- 컨테이너 종료 직전

각 단계마다 다른 훅이 존재한다.

## 5. Aware 인터페이스

PPT 11~12장은 컨테이너가 빈에게 메타정보를 알려주는 인터페이스를 다룬다.

### `BeanNameAware`

- 빈 이름을 전달받는다
- 로그, 진단, 여러 빈 구분 등에 사용 가능

### `ApplicationContextAware`

- `ApplicationContext` 자체를 전달받는다
- 다른 빈을 직접 조회하거나 컨테이너 정보를 읽을 때 사용한다

주의:

- 편리하지만, 남용하면 일반적인 DI 구조보다 결합이 커질 수 있다

## 6. 외부 설정 프로퍼티

PPT 13~14장은 설정값 분리를 설명한다.

### `@PropertySource`

- 별도 프로퍼티 파일을 명시적으로 로드

### `@Value`

- 단일 값 주입에 적합

예:

- 서버 주소
- 포트
- 간단한 메시지 값

### `@ConfigurationProperties`

- 묶여 있는 설정들을 객체 단위로 매핑

예:

- datasource
- mail
- API 설정

## 7. 슬라이드 이미지 해석

6주차 PPT의 이미지도 핵심 개념을 분명히 보여준다.

### singleton 그림

- 클라이언트 A, B, C가 모두 같은 `memberService` 인스턴스를 받는다

### prototype 그림

- 요청할 때마다 `prototypeBean x01`, `x02`, `x03`처럼 다른 객체가 생성된다

### request 그림

- 같은 컨트롤러/서비스 흐름이라도 사용자 A 요청과 B 요청이 서로 다른 request 빈을 가진다

### 서로 다른 스코프 의존 그림

- singleton이 prototype을 직접 들고 있으면 새로 생성되지 않는다
- 그래서 `ObjectFactory`처럼 지연 조회가 필요하다

## 8. 시험 대비 핵심 정리

- singleton은 기본값이며 컨테이너당 하나다.
- prototype은 요청할 때마다 새 객체를 만든다.
- request는 요청마다, session은 사용자 세션마다 유지된다.
- singleton이 prototype을 직접 주입받으면 같은 prototype이 반복 사용된다.
- `ObjectFactory`는 prototype/request 빈을 필요 시점에 안전하게 조회하는 방법이다.
- 라이프사이클 훅은 어노테이션, 인터페이스, 커스텀 메서드 세 방식으로 볼 수 있다.
- `BeanNameAware`는 빈 이름, `ApplicationContextAware`는 컨테이너 자체를 전달받는다.
- 단일 값은 `@Value`, 구조화된 설정은 `@ConfigurationProperties`가 적합하다.
